# ГлавстройLLM RAG: текущее состояние

Актуально на 22.07.2026. Этот файл содержит только технический handoff без паролей,
API-ключей, документов и извлечённых корпоративных данных. Подробный эксплуатационный
runbook находится в `README.md`.

## Цель и архитектура

Сервис разделяет постоянный корпоративный корпус и файл, приложенный к текущему чату:

```text
корпоративный корпус -> BGE-M3 -> PostgreSQL/pgvector -> корпоративный Qwen
DOCX из чата -> Open WebUI -> PUT /process -> OOXML extractor -> контекст -> корпоративный Qwen
```

Во втором маршруте Open WebUI получает текст с сохранённой Word-нумерацией, а gateway
распознаёт вложенный source-context и не запускает поверх него второй pgvector-поиск.

На Ubuntu код ожидается в `~/openwebuig`, исходные документы — в одном выбранном каталоге
(`~/rag_template/docs` либо `~/openwebuig/docs`). API работает как systemd-сервис
`document-search-rag-api`, PostgreSQL — в контейнере `document-search-postgres`, интерфейс —
в `document-search-webui`. Docker volume `open-webui` удалять нельзя: в нём пользователи,
настройки и история чатов.

## Что реализовано

- Локальный BGE-M3 через `sentence-transformers==5.5.0`, CPU-only PyTorch и стабильный
  идентификатор embedding-профиля. Модель кэшируется в процессе, параллельный inference
  ограничен семафором.
- Перед каждым семантическим поиском проверяются model id и размерность всех векторов.
  Проверка и поиск держат shared transaction advisory lock, поэтому запрос не может увидеть
  разные поколения корпуса. `/health` возвращает `200` только для непустого совместимого
  индекса, иначе `503`.
- Запрос «какие документы есть по теме» агрегирует результаты по документам, учитывает
  несколько сильных чанков и применяет абсолютные и относительный score-пороги. Слабый
  тематический хвост отсекается; запрос полного каталога по-прежнему выводит весь реестр.
- Автоматическая Word-нумерация восстанавливается из `numbering.xml`, включая наследование
  стилей, overrides, restart, Roman/Latin/Russian formats, bullets и нумерацию в таблицах.
  Видимый префикс сохраняется в полном тексте и чанках.
- Номера разделов и пунктов хранятся в отдельных полях PostgreSQL. Конкретный пункт,
  раздел или приложение извлекается структурным SQL-запросом без embedding-поиска.
- История маршрутизатора расширена до 24 сообщений и 30 000 символов. Ссылки `[2]` и
  «источник 2» разрешаются как citations, а «пункт 2» — как структура документа. Продолжения
  «оба/обе/both/all» могут выбрать несколько ранее показанных пунктов; неоднозначная ссылка
  приводит к уточнению вместо выдуманного ответа. Для раздела или приложения можно загрузить
  корень вместе со всеми подпунктами.
- Open WebUI persistent config автоматически переводится на внешний document loader как для
  legacy `config(id,data)`, так и для текущего `config(key,value)`. Перед изменением создаётся
  приватный backup `webui.db`; пользователи, LDAP, чаты и volume не удаляются.
- `/process` принимает DOCX и UTF-8 text, проверяет размер, тип и целостность, требует Bearer
  key и возвращает Open WebUI `page_content` с metadata. DOCX проходит через тот же OOXML
  extractor, который восстанавливает автоматическую нумерацию из `numbering.xml`. До разбора
  действуют потоковый лимит тела, лимиты ZIP members/распакованного размера/compression ratio;
  повреждённый пользовательский OOXML возвращает `400`, а не роняет worker.
- Gateway извлекает source-блоки Open WebUI из текущего user/system message, удаляет их из
  истории, считает содержимое файла недоверенными данными и отвечает по нему напрямую с
  citations. Старое вложение из предыдущего user-turn не перехватывает новый обычный запрос.
- Header/footer, textbox, referenced footnote/endnote и пропущенный Word-шаблонный текст
  сохраняются supplemental-блоками. Повторы дедуплицируются для поиска, а их количество и
  provenance остаются в `source_occurrences`, `source_parts` и `source_locations`.
- Независимый аудит читает OOXML напрямую стандартной библиотекой и не разделяет парсер с
  extractor. Он сверяет source hash, токены, блоки, чанки, структурные поля и строки БД.
- Полный корпус сначала собирается в отдельный кандидат с SHA-256 исходников и артефактов.
  Независимый strict-аудит создаёт связанный с `READY` маркер `AUDITED`; без него promotion
  запрещён. Индексатор сверяет точное множество DOCX, manifest и SHA-256 именно прочитанных
  bytes до embedder, повторяет проверку перед commit и заменяет корпус одной транзакцией под
  exclusive PostgreSQL advisory lock.
- `scripts/evaluate_rag.py` запускает golden-набор через OpenAI-compatible API и проверяет
  route, источники, названия, обязательные/запрещённые фразы и latency.

## Что ещё требует внешнего доступа

Текущий Ubuntu-стенд `172.19.225.124` не был изменён в этой сессии. 21.07.2026 SSH
завершался `Connection timed out during banner exchange`; HTTP на порту 80 также был
недоступен с текущей сети. Поэтому нельзя утверждать, что сервер уже получил новый `main`,
что рабочая БД переиндексирована или что Open WebUI прошёл smoke-тест.

Когда сеть станет доступна, порядок действий такой:

1. Только чтением проверить `git status`, commit, свободный диск/RAM, systemd, контейнеры,
   `/health`, число DOCX и текущие DB counts/model ids.
2. Не перезаписывать грязное дерево и не удалять volume `open-webui`.
3. Сделать приватный `pg_dump` вне checkout, затем `git pull --ff-only origin main` и
   `uv sync --frozen`; проверить локальные веса `/opt/models/bge-m3`.
4. Выбрать один authoritative `DOCS_DIR`, зафиксировать фактическое
   `EXPECTED_DOCUMENTS`, собрать новый candidate и выполнить strict pre-audit с
   `--certify-candidate`.
5. Запустить `step3_index_chunks.py --atomic --replace-corpus`; успехом считать только
   строку `Atomic commit completed`.
6. Выполнить strict post-audit, validation query, перезапуск API, `/health`, golden-набор и
   smoke-тест Open WebUI. Затем выполнить `scripts/repair_openwebui_access.sh`, проверить
   `/process` на DOCX с автоматической нумерацией и повторно загрузить такой файл в новый чат.
   Ранее сохранённые Open WebUI чанки автоматически не пересчитываются. Точные команды
   приведены в разделах 5–12 `README.md`.

До завершения этих шагов локальный код и GitHub могут быть готовы, но production deployment
остаётся неподтверждённым.
